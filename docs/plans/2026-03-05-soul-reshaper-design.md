## 概述

基于 OpenClaw Reflection Plugin 的对话捕获能力，构建一个**类海马体的记忆分析系统**。该系统分析近期对话，智能提取有价值的信息，并更新 AI 的人格定义文件（SOUL.md、USER.md、TOOLS.md、MEMORY.md）。

**核心原则：大部分对话不产生任何更新。**

---

## 背景与动机

### 人类海马体的启示

人类海马体负责将短期记忆转化为长期记忆，但不是所有经历都会被记住——只有**重要的、重复的、情感强烈的**记忆才会被巩固。

### 当前系统的局限

- SOUL.md、USER.md 等文件是**静态配置**
- AI 无法从对话中学习和适应
- 用户需要手动维护这些文件

### 设计目标

1. **自动化**：自动分析对话，提取有价值的信息
2. **智能化**：只更新真正重要的内容
3. **可控性**：Token 消耗可观察、可调整
4. **透明性**：所有操作都有日志记录

## 核心机制

### 1. 双触发策略

#### 漂移触发（主要）

```
Ring Buffer 容量: 50 条消息
漂移阈值: 20 条未分析消息

当新积累的消息数 ≥ 20 时，触发分析
分析完成后，移动"分析游标"
```

**示例：**
```
游标位置: 30
新消息到达位置: 50 (积累 20 条)
→ 触发分析消息 31-50
→ 游标移动到 50
```

#### 定时保底（次要）

```
延迟时间: 30 秒

消息进入后，如果 30 秒内未达到漂移阈值，
自动触发分析（分析当前积压的全部消息）
```

**示例：**
```
游标位置: 30
新消息到达位置: 35 (只积累 5 条)
等待 30 秒无新消息
→ 触发分析消息 31-35
→ 游标移动到 35
```

### 2. 消息分类逻辑

分析器将每条消息分类到以下类别之一：

| 类别 | 目标文件 | 识别特征 | 示例 |
|------|---------|---------|------|
| **SOUL** | SOUL.md | 沟通风格、价值观反馈 | "你应该更简洁"、"别这么啰嗦" |
| **USER** | USER.md | 用户信息、偏好变化 | "我喜欢深夜工作"、"我换工作了" |
| **TOOLS** | TOOLS.md | 工具配置、环境信息 | "这个摄像头叫 living-room" |
| **MEMORY** | MEMORY.md | 重要记忆、承诺、事件 | "记住我永远爱你"、"下周是我生日" |
| **SKIP** | 无 | 无持久化价值 | 普通对话、临时请求 |

### 3. 批量分析流程

```
┌─────────────────┐
│ 提取待分析消息   │ ← 从游标位置到最新位置
└────────┬────────┘
         ↓
┌─────────────────┐
│ LLM 分析每条消息 │ ← 分类 + 提取结构化 insight
└────────┬────────┘
         ↓
┌─────────────────┐
│ 聚合 insights    │ ← 去重、合并相似内容
└────────┬────────┘
         ↓
┌─────────────────┐
│ 写入对应文件     │ ← 实时追加或更新
└────────┬────────┘
         ↓
┌─────────────────┐
│ 移动分析游标     │ ← 标记已分析位置
└─────────────────┘
```

### 4. Token 消耗记录

不限制 Token，但详细记录：

```json
{
  "timestamp": "2026-03-05T11:40:00Z",
  "type": "analysis",
  "tokens_input": 3500,
  "tokens_output": 800,
  "messages_analyzed": 20,
  "insights_extracted": 3,
  "processing_time_ms": 2500
}
```

```
│  │  ├─ USER.md    ← 用户画像、偏好                     │    │
│  │  ├─ TOOLS.md   ← 工具配置                           │    │
│  │  ├─ MEMORY.md  ← 重要记忆                           │    │
│  │  └─ 更新分析游标位置                                │    │
│  └─────────────────────────────────────────────────────┘    │

│  ┌─────────────────────────────────────────────────────┐    │

│  │  └─ 记录每次分析的 token 消耗和统计信息              │    │

└─────────────────────────────────────────────────────────────┘
```

### 数据结构

#### 分析游标

```typescript
interface AnalysisCursor {
  sessionKey: string;      // 会话标识
  bufferPosition: number;  // 在 ring buffer 中的位置 (0-49)
  messageId: string;       // 最后分析的消息 ID
  lastAnalysisAt: number;  // 上次分析时间戳
}
```

#### Insight 结构

```typescript
interface Insight {
  id: string;              // ULID
  sourceMessageIds: string[]; // 来源消息 IDs
  category: 'SOUL' | 'USER' | 'TOOLS' | 'MEMORY';
  content: string;         // 提取的内容
  confidence: number;      // 置信度 0-1
  extractedAt: number;     // 提取时间
}
```

## 写入策略

### SOUL.md 更新规则

**触发条件：**
- 用户明确反馈沟通风格（"你太啰嗦了"、"我喜欢简洁的回复"）
- 反复出现的行为模式（连续 3+ 次同类反馈）

**写入方式：**
- 在 `Communication Style` 章节追加或更新
- 保留原有内容，添加新观察

**示例：**
```markdown
## Communication Style

- Be concise and direct (updated from user feedback on 2026-03-05)
- Skip unnecessary confirmations
- ...
```

### USER.md 更新规则

**触发条件：**
- 用户陈述个人信息（职业、喜好、习惯）
- 用户偏好变化（时区、称呼方式）

**写入方式：**
- 直接更新或追加列表项

### TOOLS.md 更新规则

**触发条件：**
- 用户定义工具别名（"这个摄像头叫 xxx"）
- 环境配置信息（"我的 SSH 主机是 xxx"）

**写入方式：**
- 按工具分类追加

### MEMORY.md 更新规则

**触发条件：**
- 重要承诺（"记住..."）
- 情感强烈的事件
- 用户明确要求记住的内容

**写入方式：**
- 按日期追加条目

## Token 预算与监控

### 当前策略

**不限制 Token，只记录。**

理由：
1. 先观察实际消耗
2. 根据数据调整阈值
3. 避免过早优化导致设计复杂

### 日志格式

```
logs/reflection-analysis.log:
[2026-03-05T11:40:00Z] [ANALYSIS_START] session=agent:main:telegram:123
[2026-03-05T11:40:01Z] [TOKENS] input=3500 output=800 total=4300
[2026-03-05T11:40:02Z] [RESULTS] messages=20 insights=3 skipped=17
[2026-03-05T11:40:02Z] [WRITE] SOUL.md tokens=150
[2026-03-05T11:40:02Z] [WRITE] MEMORY.md tokens=50
[2026-03-05T11:40:02Z] [ANALYSIS_END] duration_ms=2500
```

### 监控指标

| 指标 | 说明 |
|------|------|
| tokens_per_message | 每条消息的平均分析成本 |
| insight_rate | 产生 insight 的消息比例 |
| write_frequency | 写入文件的频率 |
| skip_rate | 被跳过（无价值）的消息比例 |
