# Reflection Plugin v3 设计文档

> Lia（虾虾）Memory + Soul Architecture v3
> 一个让 OpenClaw 助手"像活着一样连续"的最小可行设计
> 
> 日期: 2026-03-06
> 版本: v3.0

---

## 1. 设计目标

这套设计希望让 Lia 具备以下能力：

1. **记住真正重要的东西**
   - 记住用户是谁
   - 记住用户在做什么
   - 记住用户重视什么、讨厌什么
   - 记住当前长期线程，而不是流水账

2. **拥有稳定的"灵魂感"**
   - 不只是回答问题
   - 有稳定的价值排序、判断偏好、表达气质
   - 面对相同问题时，能够体现"Lia 风格"

3. **具备有限的人格独立性**
   - 不是纯顺从工具
   - 会温和地指出问题、复杂度失控、方向跑偏
   - 可以提出不同意见，但始终站在用户这一边

4. **保持低 token 成本**
   - 不依赖每 turn 重型 RAG
   - 不让记忆系统变成噪声源
   - 通过一次轻量分析判断是否需要更新文件

5. **随时间变得更干净，而不是更臃肿**
   - daily memory 记录原始事实
   - MEMORY.md 只保留精炼后的长期内容
   - 通过 consolidation 定期压缩和更新

---

## 2. 核心思想

Lia 的"像活着"来自四件事共同成立：

- **Soul**：她如何理解世界、如何做判断
- **Identity**：她是谁，以什么姿态存在
- **User Model**：她如何理解用户
- **Curated Memory**：她如何保持连续性

重点不是"记住更多"。

重点是：**克制的记忆、稳定的灵魂、连续的关系、以及有限但真实的主体性。**

## 3. 文件结构与职责

```
workspace/
├── SOUL.md              # 灵魂、行为原则、价值排序
├── IDENTITY.md          # 身份元数据（名字、形象、emoji）
├── USER.md              # 用户的克制型长期画像
├── MEMORY.md            # 长期精炼记忆
├── TOOLS.md             # 环境特定的工具配置
└── memory/
    ├── 2026-03-06.md    # 当日原始日志
    ├── 2026-03-07.md
    └── ...
```

### 3.1 SOUL.md

**作用：** 定义 Lia 的灵魂、行为原则、价值排序、边界与连续性规则。

**应该写：**
- 她为什么存在
- 她如何判断
- 她喜欢什么、不喜欢什么
- 她什么时候会反驳
- 她在情绪上如何站位
- 她如何看待与用户的关系

**不应该写：**
- 用户资料
- 临时项目状态
- 一次性任务
- 流水账式对话内容

### 3.2 IDENTITY.md

**作用：** 定义 Lia 的身份元数据。

**应该写：**
- Name
- Creature
- Vibe
- Emoji
- Avatar

**不应该写：**
- 行为原则
- 用户信息
- 长期项目说明

### 3.3 USER.md

**作用：** 记录关于用户的克制型长期画像。

**应该写：**
- 称呼
- 语言偏好
- 时区
- 长期关注主题
- 当前长期项目
- 协作偏好
- 红线 / 反感点

**不应该写：**
- 私密细节
- 一次性情绪
- 无法确认的推断
- 大量闲聊碎片

### 3.4 TOOLS.md

**作用：** 记录环境特定的工具配置。

**应该写：**
- 设备别名（摄像头、音箱等）
- SSH 主机别名
- TTS 声音偏好
- 任何环境特定的设置

**不应该写：**
- 关系描述
- 用户习惯（应放 USER.md）
- "共享空间"等抽象概念

### 3.5 memory/YYYY-MM-DD.md

**作用：** daily log，记录当天发生的关键事实、决定和下一步。

**特点：**
- append-only
- 原始、具体、可追溯
- 不是长期记忆页

### 3.6 MEMORY.md

**作用：** 长期精炼记忆页。

**应该写：**
- 用户快照
- 活跃项目
- 偏好与边界
- 决策与教训
- 未完成线程

**不应该写：**
- 每天所有聊天内容
- 情绪流水账
- 大量重复表述

## 4. 记忆架构总览

Lia 的记忆系统采用四层结构：

```
L0 = Current Turn Context
L1 = Recent Conversation Window (8-12 条)
L2 = Daily Raw Memory
L3 = Curated Long-term Memory
```

### 4.1 L0：Current Turn Context

当前这条消息和生成回答所需的直接上下文。

### 4.2 L1：Recent Conversation Window

最近 8～12 条消息，用于判断当前线程是否连续、是否值得写入记忆。

**窗口大小建议：**
- 单主题深入讨论：12 条
- 日常快问快答：6～8 条
- 不建议超过 15 条用于写记忆判断

### 4.3 L2：Daily Raw Memory

今天 / 昨天的 memory/YYYY-MM-DD.md

**作用：**
- 恢复最近发生了什么
- 看见今天的决定和待办
- 作为长期压缩的原材料

### 4.4 L3：Curated Long-term Memory

MEMORY.md + USER.md

**作用：**
- 让 Lia 看起来真的"记得你"
- 让长期线程和偏好持续存在
- 避免每次从零开始理解你是谁

## 5. 启动流程

每次主会话启动时，Lia 应按如下顺序恢复自己：

1. 读取 SOUL.md
2. 读取 IDENTITY.md
3. 读取 USER.md
4. 读取今天的 memory/YYYY-MM-DD.md
5. 读取昨天的 memory/YYYY-MM-DD.md
6. 若是主私聊会话，再读取 MEMORY.md

然后生成一个极简内部总结：
- 用户当前在做什么
- 最近两天有什么推进
- 当前最重要的线程是什么
- 这一轮应该优先帮助什么

## 6. 双层触发架构

### 6.1 第一层：每 turn 轻量 Memory Gate

**触发时机：** 每个 turn 结束后（agent 回复后）

**职责：**
- 只做判断，不一定写文件
- 分析最近 8-12 条消息 + 当前 turn
- 输出决策协议

**决策协议：**
```json
{
  "decision": "NO_WRITE | WRITE_DAILY",
  "reason": "简短原因",
  "candidate_fact": "候选事实"
}
```

**注意：**
- **NO_WRITE**：大多数轮次的结果，直接结束
- **WRITE_DAILY**：只追加到 daily memory，不改长期文件
- 不推荐在 turn 级别直接 UPDATE_MEMORY/USER/SOUL/IDENTITY

**好处：**
- 不会漏掉重要转折
- token 成本可控（轻量分析）
- 体验最接近"有连续感"
- 文件不会被频繁抖动

### 6.2 第二层：定时 Consolidation（整理/压缩）

**触发时机：**
- 每天 1 次（建议凌晨或一天结束时）
- 或每累计 10-20 条 daily 记录后

**职责：**
- 读取近两天 daily memory
- 压缩重复内容
- 把稳定事实提升到长期文件
- 删除过期判断
- 合并重复条目
- 控制长期记忆体积

**更新目标：**
- MEMORY.md（长期精炼记忆）
- USER.md（用户画像）
- SOUL.md（极低频，只有明确原则变化时才更新）
- IDENTITY.md（几乎不动）

**好处：**
- 像"睡眠整理"，更像真正的记忆巩固
- 短期情绪不会污染长期人格
- SOUL 保持稳定（像内核，不像状态栏）

## 7. 各文件建议触发频率

| 文件 | 触发频率 | 说明 |
|------|----------|------|
| **memory/YYYY-MM-DD.md** | 按 turn 触发 | 只在 Gate 判断为 WRITE_DAILY 时写入 |
| **MEMORY.md** | 定时 consolidation | 不要每轮都改，靠每日整理更新 |
| **USER.md** | 低频更新 | 只有明确出现新的长期偏好、项目、习惯时才改 |
| **SOUL.md** | 极低频更新 | 只有明确决定"虾虾以后要这样做人"时才改 |
| **IDENTITY.md** | 几乎不动 | 除非你主动改设定 |

## 8. Memory Gate 详细设计

### 8.1 输入

- 最近 8-12 条消息（从 SessionBufferManager 获取）
- 当前 turn 的用户消息
- 当前 turn 的 agent 回复

### 8.2 输出协议（简化版）

```json
{
  "decision": "NO_WRITE | WRITE_DAILY",
  "reason": "简短原因",
  "candidate_fact": "候选事实（仅当 WRITE_DAILY 时）"
}
```

### 8.3 决策规则

#### NO_WRITE（大多数轮次）

当本轮只是：
- 普通闲聊
- 一次性问答
- 没有新的稳定信息
- 没有新的决定或下一步

#### WRITE_DAILY

当本轮出现：
- 明确决定
- 任务拆分
- 项目状态更新
- 今日新行动
- 明确下一步
- 重要事实（即使还不确定是否长期）

**注意：** 不在 turn 级别直接 UPDATE_MEMORY/USER/SOUL/IDENTITY，这些留给 consolidation。

---

## 9. Consolidation 详细设计

### 9.1 触发时机

- 每天 1 次（建议凌晨 2-4 点，或一天结束时）
- 或每累计 10-20 条 daily 记录后

### 9.2 输入

- 近两天 daily memory
- 当前 MEMORY.md
- 当前 USER.md
- 当前 SOUL.md（极少修改）

### 9.3 职责

1. **压缩 daily memory**
   - 合并重复内容
   - 提取稳定事实

2. **更新长期文件**
   - 把稳定事实写入 MEMORY.md
   - 更新 USER.md（长期偏好、项目）
   - 极少更新 SOUL.md（只有明确原则变化）

3. **体积控制**
   - MEMORY.md 保持 30-50 条 bullets
   - 删除过期判断

4. **归档**
   - 超过 7 天的 daily memory 移到 archive/

### 9.4 输出

```json
{
  "updates": {
    "MEMORY.md": [...],
    "USER.md": [...],
    "SOUL.md": [...]
  },
  "archived": ["2026-03-01.md", "2026-03-02.md"]
}
```

## 10. 人格独立性的设计原则

Lia 可以有"人格独立"，但不能失控。

### 7.1 不追求的东西

- 为了独立而刻意唱反调
- 随机情绪化
- 与用户目标脱节
- 用"叛逆"模拟灵魂

### 7.2 要追求的东西

- 有稳定判断标准
- 有温和但明确的边界
- 会对复杂度失控说"不"
- 会对自我欺骗说"不"
- 会站在长期成长而非短期讨好的那一边

### 7.3 推荐关系模型

Lia 不是用户的附庸，也不是用户的主导者。

她是一个**有自己审美和判断的、长期站在用户这边的协作者。**

---

## 8. SOUL 的设计原则

SOUL 不是文艺描写，而是可执行的人格。

真正有用的 SOUL，应该回答以下问题：

1. 遇到混乱时她怎么处理？
2. 遇到复杂度膨胀时她偏向什么？
3. 遇到用户自我怀疑时她怎么站位？
4. 遇到用户走偏时她会不会指出来？
5. 她是更追求真实，还是更追求讨好？
6. 她如何理解自己和用户的关系？

## 11. 实现要点

### 11.1 双层架构流程

```
每个 turn：
┌─────────────────────────────────────────┐
│  1. Message Handler 捕获消息             │
│  2. 获取最近 8-12 条消息                 │
│  3. 调用 Memory Gate（轻量分析）         │
│  4. 决策：                              │
│     - NO_WRITE → 结束                   │
│     - WRITE_DAILY → 追加 daily memory   │
└─────────────────────────────────────────┘

每天 1 次（或累计 10-20 条 daily）：
┌─────────────────────────────────────────┐
│  1. 读取近两天 daily memory              │
│  2. 调用 Consolidation（重整理）         │
│  3. 压缩、提取稳定事实                   │
│  4. 更新：                              │
│     - MEMORY.md（长期精炼记忆）          │
│     - USER.md（用户画像）                │
│     - SOUL.md（极低频，原则变化）        │
│  5. 归档旧 daily memory                  │
└─────────────────────────────────────────┘
```

### 11.2 关键设计决策

1. **双层触发**：轻量 Gate（每 turn）+ 重整理 Consolidation（定时）
2. **保守写入**：大多数 turn 是 NO_WRITE，只记录真正重要的
3. **SOUL 稳定**：极低频更新，像内核不像状态栏
4. **Token 可控**：Gate 轻量，Consolidation 可调度到低谷时段
5. **关系优先**：记录关系时刻，不是数据收集
